# Workspace Read-Only Tools

상태: Draft v0.1  
최종 갱신: 2026-04-22

이 문서는 Personal Agent OS에서 워크스페이스를 읽기 전용으로 탐색하는 도구 계약을 정의한다.  
이 단계는 브라우저 프리뷰나 UI 렌더링이 아니라, 로컬 워크스페이스 파일 시스템을 안전하게 조회하는 런타임 문서다.

## 1. 목적

워크스페이스 도구는 에이전트가 프로젝트 구조를 이해하고, 관련 파일 내용을 읽고, 후속 작업의 근거를 확보하도록 돕는다.

핵심 원칙:

- 읽기 전용이다.
- 파일 시스템 외부로 나가지 않는다.
- 사용자 워크스페이스의 범위를 넘는 탐색을 허용하지 않는다.
- UI, 브라우저, 미리보기, 자동 실행은 포함하지 않는다.
- 결과는 후속 추론의 입력이지, 편집 권한이 아니다.

## 2. 범위 규칙

허용 범위:

- 현재 작업 워크스페이스의 파일과 디렉터리 목록 조회
- 허용된 파일의 텍스트 내용 읽기
- 경로 기반 탐색을 위한 최소한의 메타데이터 반환

제한 규칙:

- 도구는 삭제, 수정, 이동, 생성, 실행을 하지 않는다.
- 심볼릭 링크는 정책상 안전성이 확인된 경우에만 따라간다.
- 절대 경로가 주어지더라도 워크스페이스 루트 밖이면 거부한다.
- 대용량 바이너리 파일은 일반 텍스트로 읽지 않는다.
- 숨김 디렉터리와 캐시 디렉터리는 기본적으로 무시한다.

## 3. 무시 대상 디렉터리

다음 디렉터리는 기본 탐색 대상에서 제외한다.

- `.git`
- `.svn`
- `.hg`
- `.idea`
- `.vscode`
- `.next`
- `dist`
- `build`
- `out`
- `coverage`
- `node_modules`
- `Pods`
- `DerivedData`
- `vendor`
- `.cache`
- `.turbo`
- `.parcel-cache`
- `.pytest_cache`

추가 원칙:

- 운영 산출물, 캐시, 빌드 결과물은 기본적으로 읽지 않는다.
- 설정상 예외가 필요한 경우에도 정책 계층이 명시적으로 허용해야 한다.

## 4. 도구 목록

### `workspace.list_files`

워크스페이스 내 파일과 디렉터리 구조를 조회한다.

입력 계약:

```json
{
  "path": "string",
  "depth": 1,
  "include_hidden": false
}
```

- `path`: 조회 시작 경로. 워크스페이스 루트 기준 상대 경로를 사용한다.
- `depth`: 하위 탐색 깊이. 기본값은 `1`이다.
- `include_hidden`: 숨김 항목 포함 여부. 기본값은 `false`다.

출력 계약:

```json
{
  "root": "string",
  "path": "string",
  "entries": [
    {
      "name": "string",
      "path": "string",
      "type": "file | directory",
      "hidden": false,
      "size": 1234,
      "children": []
    }
  ],
  "truncated": false
}
```

- `entries`는 정렬 가능한 안정적 순서를 가져야 한다.
- `children`은 `depth` 범위 내에서만 포함한다.
- `truncated=true`면 결과가 길이 제한으로 잘린 것이다.

### `workspace.read_file`

개별 파일의 내용을 읽는다.

입력 계약:

```json
{
  "path": "string",
  "encoding": "utf-8",
  "start_line": 1,
  "end_line": 200
}
```

- `path`: 워크스페이스 루트 기준 상대 파일 경로다.
- `encoding`: 기본값은 `utf-8`이다.
- `start_line`, `end_line`: 선택적 라인 범위다. 생략 시 전체 또는 안전한 기본 범위를 읽는다.

출력 계약:

```json
{
  "path": "string",
  "encoding": "utf-8",
  "line_start": 1,
  "line_end": 200,
  "content": "string",
  "truncated": false
}
```

- 텍스트는 원문을 보존하되, 정책상 필요한 경우만 마스킹한다.
- 너무 큰 파일은 분할 읽기 또는 `truncated=true`로 응답한다.
- 바이너리 감지 시 읽기 실패로 처리한다.

## 5. 위험과 역량

역량:

- 디렉터리 구조 파악
- 관련 파일 탐색
- 구현 근거 확보
- 문서와 코드 간 불일치 확인

위험:

- 민감 파일 노출
- 경로 탈출
- 대용량 파일로 인한 성능 저하
- 잘못된 인코딩으로 인한 내용 왜곡
- 숨김 파일 또는 캐시 파일의 불필요한 오염

완화 규칙:

- 루트 밖 경로는 차단한다.
- 무시 디렉터리는 기본적으로 제외한다.
- 파일 읽기는 라인 단위 제한을 둔다.
- 실패 시 불완전한 추측보다 명시적 오류를 반환한다.

## 6. 실패 모드

### `not_found`

- 대상 경로가 존재하지 않을 때 발생한다.
- 사용 가능한 후속 조치: 상위 디렉터리 목록 재조회.

### `permission_denied`

- 워크스페이스 범위 밖이거나 정책상 금지된 경로일 때 발생한다.
- 사용 가능한 후속 조치: 경로를 워크스페이스 내부로 제한한다.

### `ignored_path`

- 무시 대상 디렉터리 또는 파일 유형일 때 발생한다.
- 사용 가능한 후속 조치: 다른 경로를 선택한다.

### `invalid_path`

- 상대 경로 규칙을 위반하거나 정규화가 실패할 때 발생한다.

### `unsupported_encoding`

- 지정 인코딩이 처리 불가하거나 파일이 텍스트가 아닐 때 발생한다.

### `too_large`

- 응답 크기 한도를 넘을 때 발생한다.
- 사용 가능한 후속 조치: 범위를 줄여 다시 읽는다.

### `timeout`

- 파일 시스템 응답 지연이 임계치를 넘을 때 발생한다.
- 사용 가능한 후속 조치: 더 작은 범위로 재시도한다.

## 7. 테스트 시나리오

- `workspace.list_files`가 루트 디렉터리에서 정상적인 파일과 디렉터리를 반환한다.
- `workspace.list_files`가 `node_modules`와 `dist`를 기본적으로 제외한다.
- `workspace.list_files`가 `depth=0` 또는 `depth=1`에서 과도한 하위 탐색을 하지 않는다.
- `workspace.list_files`가 워크스페이스 밖 경로를 거부한다.
- `workspace.read_file`이 UTF-8 텍스트 파일을 정상 반환한다.
- `workspace.read_file`이 라인 범위를 적용해 부분 읽기를 수행한다.
- `workspace.read_file`이 바이너리 파일을 거부한다.
- `workspace.read_file`이 존재하지 않는 경로에 대해 `not_found`를 반환한다.
- `workspace.read_file`이 무시 디렉터리 안 파일에 대해 `ignored_path`를 반환한다.
- 두 도구 모두 UI 미리보기나 브라우저 렌더링 단계 없이 동작해야 한다.

## 8. 구현 메모

- 이 문서는 읽기 전용 워크스페이스 도구 계약만 다룬다.
- 편집 도구, 적용 도구, diff 도구는 별도 문서로 분리한다.
- 브라우저 기반 미리보기, 화면 캡처, 시각적 검사 기능은 이 단계의 범위 밖이다.
